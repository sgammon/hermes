main Module¶

This is where it all begins... ah, the wonderful main.py. From here, we stitch together apptools and gevent to provide an entrypoint for apptools.dispatch.

In addition to being the WSGI entrypoint for AppFactory (and any other WSGI-compliant platform), this is also where you run the devserver...

Running the devserver¶

Only one WSGI application can be run at a time, and running Hermes can easily be done from the command line.

From the root of your project (the folder outside app/):

$ tools/devserver

==== Hermes Devserver ====
Preloaded module bundle: "apptools".
Preloaded module bundle: "tracker".
Failed to preload compiled templates.
Preloaded module bundle: "hermes".
[TRACKER]: Datastore: Initialized datastore engine.
[TRACKER]: Debug mode is ON, serving on port 8080.

!! Started listener for app <components.tracker.EventTracker> on host/port :8080. !!

If you’d like to start up on port 80 instead:

$ sudo tools/devserver --port=80

Note

For ports under 1024, Unix/Linux requires the use of sudo.

Dispatching in production¶

Many platforms, including AppFactory and Google App Engine, dispatch Python applications over WSGI. This app’s main module can be used with WSGI - applications are exported at main.EventTracker and main.APIServer for EventTracker and Hermes, respectively.

Here’s a sample app.yaml file for AppFactory:

appfactory:
  dispatch: main:APIServer
  processes: 2
  threads: 20

As you might imagine, the above will start 2 app processes with 20 dedicated app threads each. Apps hosted on gevent are just as easy:

appfactory:
    dispatch: main:EventTracker
    runtime: gevent
    greenlets: 1000
    processes: 1

Note that AppFactory will monkey-patch the CPython standard library for you when you use the gevent runtime. To prevent this behavior, simply append monkey: off.

Note

Multiple processes with gevent enabled is not recommended. In many cases, you can actually achieve better concurrency using Greenlets in a single process only, as the native OS management overhead tends to outweigh performance benefits significantly.

Just for good measure, here is an AppEngine-compatible app.yaml file (you can also specify an appfactory.yaml file to avoid compatibility issues and run on both platforms!):

application: my-cool-app
version: docs-sample

runtime: python27
api_version: 1
threadsafe: yes

handlers:

- url: /v1
  script: main.APIServer

- url: /.*
  script: main.EventTracker

To be clear, those module endpoints are completely WSGI-compliant, meaning you could also simply import them and run them however you want, even from a regular Python shell.

main.devserver(app, args, port=8080, host='')[source]¶

Start a local listener for development, using gevent.pywsgi.

Parameters:
  • app – WSGI application to run. Defaults to apptools.dispatch.gateway.
  • port – Port to run the application on. Defaults to the value of config._DEVSERVER_PORT, which is usually set to 8080.
  • host – IP to bind the devserver to. Defaults to the value of config._DEVSERVER_HOST, which is usually set to 127.0.0.1. Setting this value to an empty string binds to all available IPs - make sure it is not set to this in production.

Note

Dispatching this function from Python (or the command line) will block forever via pyuwsgi.WSGIServer.serve_forever().
